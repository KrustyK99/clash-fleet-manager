Clash Fleet Manager Architecture Guide

1Executive orientation

The present system is intentionally simple: a modular browser application and a FastAPI backend are packaged together in one Docker image. On the NAS, the application code is replaceable, while the live JSON data remains outside the container in a persistent mounted folder.

The project is now in a sensible observation period. Automated testing gives confidence in the behaviours that were explicitly exercised. Daily use provides a different kind of assurance: it can reveal unusual data combinations, calculations, fields, or workflows that were not included in the formal test suite.

This document focuses on two architectural blind spots:

Docker

How the runtime is manufactured

Which files create the image, where container variables come from, how ports and folders are connected, and what changes require a rebuild.

Testing

How environments are assembled

Why PHP, FastAPI, local processes, mounted folders, packaged containers, and NAS smoke tests all exist without being pointless duplicates.

Current architectural endpoint: FastAPI + JSON is a complete and reasonable personal-application target. MariaDB and beyond-gate work can remain deferred unless the product’s needs materially change.

2The foundational Docker model

Most Docker confusion disappears once four concepts are kept separate.

→
DockerfileThe manufacturing recipe
→
Docker imageA packaged, frozen application snapshot
→
ContainerA running instance configured with ports, variables and mounts
ConceptExecutive interpretationClash Fleet Manager example
DockerfileManufacturing instructionsInstall Python dependencies, copy backend and frontend files, start Uvicorn.

A container is not the image. The image is the packaged product. A container is one running copy of that product, connected to a particular network port and data folder.

3Following a command: from Enter to a running application

Presentation purpose: this section follows npm run deploy:nas:fastapi as one continuous relay. It shows which component takes control, what artifact it produces, and when build-time and runtime configuration enter the process.

3.1 The command is a relay, not a single program

When the deployment command is entered, npm does not build the container itself. It opens package.json, finds the script definition, and launches the PowerShell deployment script. From there, each participant performs one responsibility and hands control to the next.

3.2 Step 1 — npm resolves the friendly command

npm is the launcher. It reads package.json and determines which PowerShell script to execute and which arguments to pass. At this point, Docker, FastAPI and the NAS have not yet done anything.

Human-friendly command
→
Deployment script invocation

3.3 Step 2 — PowerShell becomes the deployment orchestrator

The PowerShell script understands the infrastructure-specific details: NAS address, SSH port, image name, container name, host and container ports, NAS data directory, restart policy and optional skip flags.

It does not know the business meaning of builders, timers or account views. Its responsibility is narrower: package this application, transport it, and start it with the requested connections.

The script asks Docker to manufacture an image. During the local Compose build, three repository files cooperate:

3.5 What the Dockerfile puts into the image

1. Start with python:3.13-slimA small Linux base with Python already installed
2. Establish default environment variablesJSON store, /data, application serving and /app
3. Install container Python requirementsDependencies from backend/requirements-container.txt
4. Copy backend and frontend files into /appFastAPI code, HTML, CSS, data map and modular app-*.js files

3.6 Step 4 — The image exists, but nothing is running yet

The completed image is an immutable application snapshot. It contains the backend, frontend and installed dependencies, but it is not yet a running application. A container must later be created from it.

Image versus container: an image is the packaged deliverable; a container is a configured, running instance of that image.

3.7 Step 5 — Docker saves the image as a portable archive

The deployment script asks Docker to serialize the image into a .tar file. This archive is a generated build artifact used for transport. It is not source code and does not belong in Git.

3.8 Step 6 — SCP transfers the archive to the NAS

PowerShell uses SCP to copy the image archive and the generated remote deployment instructions to the Synology NAS. At this stage, the artifact has moved machines, but the new application is still not running.

3.9 Step 7 — Docker on the NAS loads the image

The NAS Docker engine imports the archive into its local image store. This makes the same application package available on the NAS. Loading an image still does not start it.

Architecture Principle — Build Once, Configure Everywhere. The laptop exports a portable image archive and transfers it to the NAS. Docker on the NAS reconstructs that image locally and then instantiates a brand-new container using NAS-specific runtime configuration such as port mappings, mounted folders, environment variables and restart policy. The running container never crosses the network.

3.10 Step 8 — Docker creates and starts the container

The remote deployment command creates a container from the loaded image. This is the point at which the environment-specific runtime configuration is attached:

  • the container name;
  • environment variables such as FLEET_STORE_BACKEND=json;
  • the host-to-container port mapping;
  • the NAS data-folder mount;
  • the restart policy.
Critical distinction: application files are selected at image build time. Ports, mounted folders and final runtime variables are attached when the container is started.

3.11 Step 9 — The persistent data folder is mounted

Docker maps the real NAS directory to /data inside the container:

The production data therefore survives removal and replacement of the container because it never resides solely in the container’s disposable filesystem.

3.12 Step 10 — The network port is mapped

The application listens on port 8001 inside the container, while the NAS exposes it through host port 8004:

→

3.13 Step 11 — FastAPI starts

Uvicorn launches FastAPI using the startup command stored in the image. FastAPI reads its runtime variables, sees the mounted /data directory, finds the application files under /app, and begins listening for HTTP requests.

3.14 Step 12 — The browser receives the application

The browser connects to the NAS. FastAPI returns index.html, CSS and JavaScript. The running frontend then calls API endpoints such as /api/load; FastAPI reads the mounted JSON files and returns the response.

3.15 Which repository state is deployed?

A normal build packages the files currently present in the working folder. Docker does not automatically fetch GitHub and does not require a commit. Committed changes and uncommitted working changes can both enter the image if they are included by the Dockerfile and not excluded by .dockerignore.

Governance interpretation: “deploy the latest repo” means “build from the current permitted working tree, transfer that exact image, and start a new container from it.” The image provides consistency between the laptop build and the NAS runtime.

4How a container is configured

The image contains defaults, but the running container receives the final operational configuration. Values can be supplied at several levels, with runtime values able to override image defaults.

Architecture Principle #1 — Build Once, Configure Everywhere. Build the application once into an immutable image. Deploy that same image into different environments by supplying different runtime configuration (ports, mounted folders, environment variables, restart policies and networking). The software package remains the same; only its operating environment changes.
Illustration 4.1 — Configuration layers
Dockerfile defaultsPortable assumptions built into the image
Compose or docker run valuesEnvironment-specific runtime choices
Script parametersOperator-selected NAS, ports, names, folders and optional behaviour

4.1 Application behaviour variables

VariableMeaningCurrent production value
FLEET_STORE_BACKENDSelects the persistence implementation.json
FLEET_DATA_DIRTells FastAPI where the persistence files are visible inside the container./data
FLEET_SERVE_APPEnables serving the browser application from FastAPI.1
FLEET_APP_DIRTells FastAPI where the frontend files live./app

4.2 Runtime wiring

Environment variables are only one type of container configuration. The runtime also defines:

  • Container name: how Docker identifies the running instance.
  • Port mapping: which host port forwards traffic to the application’s internal port.
  • Volume mount: which external folder appears at /data.
  • Restart policy: whether Docker should restart the container under defined conditions.
docker run -d \
  --name clash-fleet-manager-fastapi-json-production-candidate \
  -p 8004:8001 \
  -e FLEET_STORE_BACKEND=json \
  -e FLEET_DATA_DIR=/data \
  -e FLEET_SERVE_APP=1 \
  -e FLEET_APP_DIR=/app \
  -v /volume1/web/clash-timers/data:/data \
  --restart no \
  clash-fleet-manager-fastapi-json:local

The deployment script’s defaults are shown. Parameters can override the container name, data directory, host port, container port and restart policy.

5Baked files versus mounted files

This is the most important practical distinction in the present deployment.

Baked into image

Application files

  • FastAPI backend
  • index.html
  • styles.css
  • coc-data-map.js
  • All matching app-*.js files
  • Installed Python packages

Changing these requires rebuilding and redeploying the image.

Mounted at runtime

Production data

  • timers.json
  • account_views.json
  • Generated backups under the live data path

These survive container replacement because they live outside the container.

Replacement model: the container is disposable; the data is not. Deployment removes and recreates the container while reconnecting the new instance to the same live NAS data directory.

6.1 Why the frontend and API share one container

FastAPI serves both the browser files and API endpoints from the same origin. That reduces operational components and avoids the need to coordinate cross-origin browser permissions between separate frontend and backend servers.

6.2 What happens during NAS deployment

  1. The laptop builds the current image unless -SkipBuild is used.
  2. Docker saves the image into a .tar archive unless -SkipSave is used.
  3. SCP copies the archive and a generated remote shell script to the NAS.
  4. The NAS loads the image into its local Docker image store.
  5. The script validates the data directory and expected JSON files.
  6. The existing named candidate container is stopped and removed.
  7. A fresh container is started with variables, port mapping and the live data mount.
  8. The script runs smoke tests against the deployed URL and API endpoints.

7The testing architecture

The project has a testing ladder rather than one universal test. Each level protects a different seam and carries a different cost.

8PHP E2E command trace

The default npm run test:e2e delegates to the PHP E2E path. The more explicit verification command is npm run verify:php.

Illustration 8.1 — PHP test environment
Current repo filesFrontend + api.php
→
tests/runtime-appFresh disposable copy + fixture JSON
→
Stock PHP containerFolder mounted at /app
→
PlaywrightBrowser tests at port 8011

8.1 Sequence

  1. tests/support/prepare-test-app.mjs deletes any old tests/runtime-app.
  2. It copies the current frontend modules and api.php into that folder.
  3. It copies known fixture versions of timers.json and account_views.json.
  4. Playwright’s webServer starts npm run serve:test:docker.
  5. That command launches the stock php:8.3-cli image.
  6. The disposable folder is mounted from Windows into container path /app.
  7. PHP’s development server listens inside the container and host port 8011 forwards to it.
  8. Playwright runs desktop and applicable mobile-browser tests.
  9. When Playwright ends, its managed test server is stopped; the PHP container uses --rm, so Docker removes it.
Why a mount here? The PHP test uses an unmodified stock PHP image. The runtime application is supplied through a mounted disposable folder, avoiding the need to build a special PHP image for every test run.

9FastAPI E2E command trace

npm run verify:fastapi:e2e deliberately forces Playwright not to reuse an existing server. This ensures the test runner creates and controls a fresh FastAPI test process.

Illustration 9.1 — FastAPI test environment
Current backend codeRuns from repository root
+
tests/runtime-appDisposable frontend + fixture data
→
Local Python / Uvicorn processNot a Docker container
→
PlaywrightBrowser tests at port 8001

9.1 Sequence

  1. The command sets APP_E2E_TARGET=fastapi and PLAYWRIGHT_REUSE_EXISTING_SERVER=0.
  2. Playwright identifies FastAPI as the application target and uses base URL http://127.0.0.1:8001.
  3. The disposable test application is rebuilt from current frontend files and fixtures.
  4. serve-fastapi-test-app.mjs selects the local virtual-environment Python when available.
  5. It launches Uvicorn from the repository root using current backend code.
  6. It sets FLEET_APP_DIR to the disposable frontend folder and FLEET_DATA_DIR to its fixture-data subfolder.
  7. Playwright executes the browser suite.
  8. Playwright terminates its managed server; the Node launcher forwards the termination signal to Uvicorn.
Important correction to the simplest mental model: the standard FastAPI E2E path does not create a temporary FastAPI Docker container. It creates a temporary FastAPI process and disposable runtime environment. Container packaging is tested separately.

11.1 The practical management question at every seam

For each connection, ask:

  1. What crosses this boundary?
  2. Which file or script controls it?
  3. How is the boundary tested?
  4. What survives when one side is replaced?

This provides architectural control without needing to memorize every implementation detail.

13What to remember

  1. A Dockerfile builds an image; Compose or docker run starts a container.
  2. The image is a frozen snapshot of permitted working-tree files at build time.
  3. Application code is baked into the production image; live JSON data is mounted from the NAS.
  4. Replacing the container does not replace the mounted data.
  5. The standard PHP E2E path uses a temporary Docker container and a mounted disposable app folder.
  6. The standard FastAPI E2E path uses a temporary local Python process, not a Docker container.
  7. Container verification exists to test packaging, which application E2E tests alone cannot prove.
  8. The NAS deployment script owns the laptop-to-NAS seam: build, transfer, runtime wiring and smoke testing.
  9. Daily use is now a valid final test layer for behaviours that were never explicitly encoded in automated tests.
Executive summary: the architecture separates replaceable application packaging from persistent business data, and it applies progressively broader tests across the main seams. You do not need intimate code knowledge to manage it responsibly; you need to know which artifact controls each boundary and which test proves that boundary.
